# 智能仓储管理系统 技术规格说明书



## 第一章 系统概述



### 1.1 背景与目标

随着电商行业的高速发展，传统仓储管理模式已无法满足日益增长的订单处理需求。深圳星辰科技有限公司基于多年仓储行业经验，自主研发了智能仓储管理系统（Intelligent Warehouse Management System，简称 IWMS），旨在通过数字化手段全面提升仓储运营效率。

该系统于2022年启动立项，历时1 8个月完成核心功能开发，2 0 2 4年1月正式发布V2.0版本。截至2 0 2 4年6月，系统已在全国2 3个仓储中心成功部署，累计处理订单超过4 7 0 0万单，系统可用性达到9 9 . 9 7%。

系统建设的核心目标有三个方面。第一，将人工拣货效率提升至少6 0%，通过智能路径规划减少拣货员的无效行走距离。第二，将库存准确率从行业平均的9 6 . 2%提升至9 9 . 5%以上，通过实时盘点和异常预警消除库存差异。第三，将订单处理周期从平均4 . 2小时压缩至1 . 5小时以内，满足当日达和次日达的履约承诺。

### 1.2 系统架构

IWMS采用微服务架构设计，整体分为三层：接入层、业务层和数据层。

接入层负责处理来自外部系统的请求，包括电商平台订单推送、ERP系统数据同步以及移动终端（PDA/叉车终端）的操作指令。接入层通过API网关统一管理鉴权、限流和路由，峰值处理能力为每秒8 0 0 0个并发请求。

业务层是系统的核心，由1 7个微服务组成，涵盖入库管理、库位管理、拣货调度、出库管理、盘点管理等主要业务域。各微服务通过Kafka消息队列进行异步通信，服务间调用采用gRPC协议，平均接口响应时间为2 3毫秒。

数据层采用混合存储策略：热数据（近3 0天操作记录）存储在MySQL集群，冷数据归档至HDFS分布式文件系统，实时计算数据存储在Redis集群，搜索索引数据存储在Elasticsearch。

## 第二章 核心功能模块



### 2.1 智能入库管理

#### 2.1.1 收货预约与排期

仓库收货能力是有限的。系统通过收货预约模块对到货时间进行精细化管控，防止多批次货物同时到达导致月台拥堵。

供应商在发货前2 4小时内，须通过供应商门户提交收货预约申请，填写预计到货时间、货物品类、托盘数量以及车牌号等信息。系统将根据当日月台使用率、已预约排期和货物优先级，自动分配月台号和到货时间窗口（每个时间窗口为2小时）。

若预约时间段内月台占用率超过8 5%，系统将自动推荐相邻时间段，并通过短信和邮件通知供应商。紧急收货（如生产缺料）可通过ERP系统发起优先级调整申请，经仓库主管审批后系统自动插队排期。

#### 2.1.2 到货验收流程

货物到达月台后，系统触发验收工作流。验收员通过手持PDA扫描货物条码或托盘码，系统自动比对采购订单信息，核验品名、规格、数量是否与订单一致。

验收差异处理规则如下：数量短缺超过3%触发自动拒收流程；数量溢出在5%以内自动接收并生成差异记录；品质异常由验收员拍照记录，系统自动推送至质检部门处理。验收完成后，系统平均在8秒内生成入库单并推送至WMS库位分配模块。


### 2.2 库位管理与货物定位

#### 2.2.1 库位编码规则

系统采用四级编码体系对库位进行唯一标识：仓库代码（2位）+ 库区代码（2位）+ 货架编号（3位）+ 层位号（2位）。

例如，库位编码"SZ-A1-023-03"表示：深圳仓（SZ）、A1库区、第023号货架、第3层。该编码规则支持最多99个仓库、99个库区、999组货架和99个层位，能够满足超大型仓储中心的使用需求。

#### 2.2.2 智能库位分配算法

库位分配是影响拣货效率的核心因素。系统采用基于销售频次的动态分区策略（ABC分区法），将货物按周销售频次分为三类：

A类商品（周销售频次＞5 0 0次）占库存品类的约1 8%，但贡献了约7 2%的出库量。这类商品被优先分配至距离出库口最近的黄金区域（距出库口0-1 5米范围内），以最大化拣货效率。

B类商品（周销售频次1 0 0-5 0 0次）占库存品类的约3 5%，贡献约2 2%的出库量。这类商品分配至中间区域（距出库口1 5-4 0米范围内）。

C类商品（周销售频次＜1 0 0次）占库存品类的约4 7%，仅贡献约6%的出库量。这类商品分配至距离出库口最远的区域，充分利用深库位资源。

系统每周日凌晨2:00自动运行ABC重分类计算，对分类变动的商品生成移位建议单。移位作业在业务低峰期（每日02:00-06:00）由系统调度执行，平均每次重分类涉及商品数量约1 2 0 0个SKU。

### 2.3 拣货调度系统

#### 2.3.1 拣货任务生成规则

当订单进入拣货队列后，系统并非立即为每张订单生成独立拣货任务。这样做会导致拣货员重复路过同一货架区域，造成极大的路径浪费。

系统采用波次合并策略，将时间窗口内（默认1 5分钟）的待拣订单进行集单处理。集单规则综合考虑四个维度：订单优先级（VIP订单、加急订单优先）、配送时效要求（当日达优于次日达）、货物库区分布（同一库区的订单优先合并）、拣货员当前位置（就近分配）。

每个波次最多合并2 0 0张订单，生成后系统采用旅行商算法（TSP近似解）规划最优拣货路径，确保拣货员沿单一方向顺序作业，避免折返。实测数据显示，波次合并策略使拣货员单次任务行走距离从平均1 . 8公里降低至0 . 9公里，效率提升约5 0%。

#### 2.3.2 异常处理机制

拣货过程中常见的异常情况有三类。

第一类是库位缺货异常。当拣货员扫描库位后系统提示无货，但库存记录显示有库存时，系统立即触发库存异常工单，同时自动在其他库位中搜索相同商品的备用库存。若找到备用库位，系统推送新库位信息至拣货员PDA；若无备用库存，订单自动进入缺货处理队列，由客服通知买家。

第二类是商品损坏异常。拣货员发现商品破损时，通过PDA拍照上传，系统自动生成次品下架工单并从库存中扣减，同时在订单中搜索可替代的完好商品。

第三类是设备故障异常。PDA离线或叉车终端故障时，系统检测到心跳超时（默认3 0秒）后，自动将该设备的未完成任务重新分配给其他在线设备，保证作业连续性。

### 3.2 数据安全与备份策略

系统采用三层数据保护机制。

第一层是实时主从同步。MySQL采用半同步复制模式，主库写入的数据在至少一个从库确认接收后才返回成功响应。主从复制延迟在正常网络条件下不超过1 0 0毫秒，从库可在主库故障后3 0秒内自动完成主从切换。

第二层是定时全量备份。每日凌晨3:00对全量数据进行备份，备份文件加密后存储至异地灾备中心（深圳→上海），备份完成后系统自动验证备份文件完整性。全量备份数据保留周期为9 0天。

第三层是操作日志归档。所有数据变更操作记录详细的操作日志（操作人、操作时间、变更前后的值），日志文件实时同步至HDFS，支持按时间点进行数据恢复（RPO≤5分钟，RTO≤3 0分钟）。


## 第四章 集成接口规范

### 4.1 API 认证机制

所有外部系统调用IWMS的API，均须通过OAuth 2.0客户端凭证模式（Client Credentials Flow）进行身份认证。

认证流程如下：首先，接入方使用分配的client_id和client_secret，向认证服务器（https://auth.iwms.starctech.com/oauth/token）发起POST请求，获取access_token。access_token有效期为7 2 0 0秒（2小时），过期前3 0 0秒系统主动推送刷新通知。其次，所有API请求须在HTTP Header中携带"Authorization: Bearer {access_token}"字段。未携带或token失效的请求将返回HTTP 4 0 1状态码。

IP白名单是认证的第二道防线。每个接入方最多配置1 0个白名单IP，来自非白名单IP的请求即使携带有效token也会被拦截，返回HTTP 4 0 3状态码。

- include_locked（可选）：是否包含锁定库存，默认false

## 第五章 常见问题解答（FAQ）

Q：系统支持哪些条码标准？

A：系统支持以下条码标准：EAN-13（商品标准条码）、EAN-8（短条码）、Code-128（物流条码，支持字母数字混合）、QR码（二维码，用于托盘和货位标识）、DataMatrix（高密度二维码，用于小尺寸标签）。对于企业自定义条码，可通过"条码规则配置"功能进行扩展，最长支持6 4位自定义编码。

Q：PDA设备离线后数据会丢失吗？

A：不会丢失。系统为每台PDA配置了本地缓存队列（容量为1 0 0 0条操作记录）。当网络断开时，操作数据暂存于本地缓存；网络恢复后，系统自动按时间顺序将缓存数据同步至服务器，同步完成后清空本地缓存。若离线时间超过4小时或缓存队列已满，PDA将发出警报提示操作员寻找网络恢复点。

Q：如何处理商品被多个订单同时锁定的情况？

A：系统采用乐观锁机制处理库存并发竞争。当多个订单同时尝试锁定同一商品的库存时，系统按订单进入时间顺序（毫秒级时间戳）依次处理，先到先得。后续订单若库存不足，系统将自动触发缺货处理流程：优先从在途库存中预配，若无在途库存，则通知客服进行订单拆分或取消处理。

## 附录A：术语表

SKU（Stock Keeping Unit）：库存管理单元，是区分商品不同规格、颜色、尺寸的最小单位。例如，同一款T恤的红色XL码和蓝色L码分别是两个不同的SKU。

波次（Wave）：将一定时间窗口内的多张拣货订单合并为一批，由一名或多名拣货员协同完成的作业单元。波次合并是提升拣货效率的核心手段。

ABC分区：根据商品销售频次将仓库库位分为A（高频）、B（中频）、C（低频）三个区域，将高频商品放置在最易到达的位置，以减少拣货行走距离。

RPO（Recovery Point Objective）：恢复点目标，指系统发生故障后，可接受的最大数据丢失时间范围。本系统RPO为5分钟，即故障发生前5分钟内的数据可被完整恢复。

RTO（Recovery Time Objective）：恢复时间目标，指系统从故障发生到恢复正常运行所允许的最长时间。本系统RTO为3 0分钟。
